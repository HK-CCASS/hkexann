## 🔄 断点续传流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant Main as ManualHistoricalBackfillProcessor
    participant Resume as 断点续传管理器
    participant State as 状态文件
    participant Hist as CorrectedHistoricalProcessor
    participant Callback as 进度回调
    
    User->>Main: 运行脚本
    Main->>Resume: 检查断点状态
    Resume->>State: 读取 .manual_backfill_resume.json
    
    alt 发现未完成任务
        State-->>Resume: 返回状态（已处理123只）
        Resume->>Resume: 验证参数匹配
        Resume->>Resume: 计算剩余股票（2229只）
        Resume-->>Main: 续传模式: remaining_stocks
        Note over Main: original_total_stocks = 2352<br/>resumed_from = 123
    else 无断点或参数不匹配
        Resume-->>Main: 新任务模式
        Note over Main: 从头开始
    end
    
    Main->>State: 保存初始状态
    Main->>Hist: 设置 monitored_stocks = remaining_stocks
    Main->>Hist: process_historical_announcements()
    
    loop 处理每批股票（5只/批）
        loop 处理每只股票
            Hist->>Hist: 查询历史公告
            Hist->>Hist: 过滤相关公告
            Hist->>Hist: 下载PDF
            Hist->>Hist: 向量化处理
            Hist->>Hist: 存储到Milvus + ClickHouse
            
            Note over Hist: ✅ 单只股票完成
            
            Hist->>Hist: 更新 processed_stocks.append(stock_code)
            Hist->>Hist: 更新 processing_stats
            Hist->>Callback: 调用 progress_callback(progress_data)
            Callback->>Resume: _resume_state_callback
            Resume->>Resume: 更新进度条
            Resume->>State: 保存状态（实时）
            
            Note over State: processed_stocks: [00738, 01440, ...]<br/>stats: {processed: 125, ...}
        end
        
        Note over Hist: 批次间休息 2秒
    end
    
    Hist-->>Main: 返回处理结果
    Main->>Resume: mark_resume_completed()
    Resume->>State: status = "completed"
    Main-->>User: 显示处理统计
```

    开始处理 --> 批次1: 股票124-128
    批次1 --> 股票124: 查询+下载+向量化

    保存进度124 --> 股票125
    股票125 --> 保存进度125
    保存进度125 --> 股票126
    股票126 --> 保存进度126
    保存进度126 --> 股票127
    股票127 --> 保存进度127
    保存进度127 --> 股票128
    股票128 --> 保存进度128

## 📈 性能指标

| 指标 | 单位 | 说明 |
|------|------|------|
| 批次大小 | 5只/批 | 平衡并发和内存 |

## 📝 架构设计原则

1. **单一职责**：每个组件负责特定功能
   - `ManualHistoricalBackfillProcessor`：总控和断点续传
   - `CorrectedHistoricalProcessor`：批量处理逻辑
   - `RealtimeDownloaderWrapper`：下载和分类
   - `RealtimeVectorProcessor`：向量化处理

2. **容错设计**：
   - 多层重试机制（ClickHouse 3次，下载器内置重试）
   - 失败股票标记，避免重复尝试
   - 降级处理（批量失败转单个更新）

3. **可观测性**：
   - 详细的进度日志
   - 实时进度条
   - 完整的统计信息
   - 状态文件可检查

4. **性能优化**：
   - 批量处理减少API调用
   - 并发控制避免过载
   - 连接复用减少开销
   - 分段查询避免超时
